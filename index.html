1.1 项目与项目管理的本质

"为什么"：信息技术项目管理的危机

我们首先需要理解这门学科的迫切性。课程导论部分呈现了关于IT项目失败的严峻统计数据：平均成功率仅为16%，高达50%的项目需要补救，而34%的项目则彻底失败。这些项目平均超出预定时间222%，实际成本是估算成本的189%。项目失败的主要原因被归结为两点：软件项目日益增长的复杂性，以及合格的软件项目管理人才（即项目经理）的严重缺乏。

项目失败十大因素（节选）

这种现象揭示了一个核心问题：软件开发的独特性质使其本质上难以管理。若缺乏一套系统化的管理框架，项目极易陷入混乱，导致时间和资源的巨大浪费。因此，软件项目管理并非一种可选的管理风格，而是应对行业固有挑战、确保项目成功的必要科学与艺术。

定义"项目"

"项目"是为创造特定产品、服务或成果而进行的临时性工作。它与组织的日常"运营"形成鲜明对比，运营是重复和持续的，而项目是独特和临时的。其核心特征包括：

  • 明确的目标 (Clear Objectives): 项目存在的理由，也是衡量其成功的最终标准。
  • 独特性 (Uniqueness): 没有两个项目是完全相同的，总会带来独特的挑战。
  • 时限性 (Temporariness/Time-bound): 有明确的开始和结束日期。
  • 不确定性 (Uncertainty): 充满风险，需求可能变更，技术可能出现问题。
  • 结果的不可逆转性 (Irreversibility of Results): 项目产出一旦交付，修改或撤销的成本极高。

"软件项目"的独特性质

软件项目不仅继承了一般项目的特征，还因其内在属性带来了额外的复杂性。这些特性是理解其管理挑战的关键：

  • 技术发展快 (Rapid Technological Change)
  • 经验失效快 (Rapid Obsolescence of Experience)
  • 复杂度高 (High Complexity)
  • 主要依靠脑力劳动 (Primarily Mental Labor)
  • 过程可见性差 (Poor Process Visibility): 这是最核心的挑战之一。你无法像观察一座正在建造的桥梁那样"看到"软件的开发进度，这使得管理者难以直观评估项目的真实状态。
  • 结果难于测试 (Difficult to Test Results)

1.2 项目管理宏观框架

项目管理过程框架

五大过程组（项目生命周期）

  • 启动过程 (Initiating): 正式授权项目，制定项目章程，任命项目经理。
  • 规划过程 (Planning): 定义目标和行动路线，产出范围说明书、WBS、进度计划、预算等。
  • 执行过程 (Executing): 整合资源执行计划，产出项目的可交付成果。
  • 监控过程 (Monitoring & Controlling): 跟踪、审查和调整，识别偏差并采取纠正措施。
  • 收尾过程 (Closing): 正式完成项目，获得客户验收，总结经验教训。

PMBOK知识体系架构

项目管理知识体系指南(PMBOK)将项目管理分为十大知识领域，这些领域在整体管理的统领下，通过范围管理建立项目边界，形成了一个完整的管理框架。

十大知识领域详解

这些是项目管理的核心学科领域，需要在整个项目生命周期中持续关注和管理。

1. 整体管理: 协调和统一所有其他知识领域。
2. 范围管理: 确保项目做且只做所有必需的工作。
3. 时间管理: 管理项目按时完成。
4. 成本管理: 在批准的预算内完成项目。
5. 质量管理: 确保项目满足相关质量标准。
6. 人力资源管理: 组织、管理和领导项目团队。
7. 沟通管理: 确保项目信息及时、恰当地生成和发布。
8. 风险管理: 识别、分析、应对和监控项目风险。
9. 采购管理: 从组织外部采购所需产品或服务。
10. 干系人管理: 识别干系人，分析其期望并制定管理策略。

1.3 软件工程与项目管理的关系

理解这两者之间的关系至关重要。现代软件开发要求将这两种学科紧密地 结合 起来。

软件工程 (Software Engineering)

关注的是 产品 本身（即"做什么"），涵盖需求、设计、编码、测试等技术活动。
目标：正确地把事情做好 (Build the thing right)。

软件项目管理 (Software Project Management)

关注的是 过程 （即"如何做"），涉及规划、组织、领导和控制所有资源。
目标：做正确的事 (Build the right thing)。

项目管理三角形（铁三角）

2.1 定义工作：项目范围管理

项目范围管理是确保项目成功的基础，核心任务是明确"做什么"和"不做什么"。失败的项目中，不充分的需求规范是首要原因。

核心： 避免"镀金 (gold-plating)"——即在未经客户要求的情况下，为产品添加超出范围的功能。

范围管理的核心过程

  • 范围管理规划: 制定如何定义、核实、控制范围的计划。
  • 需求收集: 通过访谈、研讨会、原型法等技术收集需求。
  • 范围定义: 详细描述项目和产品，产出关键的项目范围说明书，明确包含和不包含的内容。
  • 创建WBS: 将项目工作分解为更小的、更易于管理的部分。
  • 范围核实: 与干系人一起正式验收已完成的可交付成果。
  • 范围控制: 监督项目范围状态，管理范围基准的变更，防止范围蔓延。

工作分解结构 (WBS)

WBS是对项目团队为实现项目目标、交付所需成果而执行的全部工作范围的、面向可交付成果的层级分解。它是整个项目规划的基石，连接了"做什么"（范围）与"做多久"（时间）和"花多少钱"（成本）的桥梁。

100%规则: WBS必须包含项目范围内的100%的工作，不多也不少。WBS的最底层单元被称为 工作包 (Work Package)，它应足够小以便可以被可靠地估算和分配。

2.2 掌握进度：项目时间管理

活动排序：四种依赖关系 (PDM)

项目活动之间存在逻辑依赖关系，最常用的紧前关系绘图法 (PDM) 定义了四种：

  • 完成到开始 (Finish-to-Start, FS): 前置活动必须完成后，后续活动才能开始（最常见）。
  • 完成到完成 (Finish-to-Finish, FF): 前置活动必须完成后，后续活动才能完成。
  • 开始到开始 (Start-to-Start, SS): 前置活动必须开始后，后续活动才能开始。
  • 开始到完成 (Start-to-Finish, SF): 前置活动必须开始后，后续活动才能完成（最罕见）。

三点估算公式 (PERT)

这是考试中的一个关键计算点。它通过综合考虑最乐观时间 ($T_O$)、最可能时间 ($T_M$) 和最悲观时间 ($T_P$) 来得出一个更可靠的期望历时 ($T_E$)。其计算公式为：

$$T_E = \frac{T_O + 4T_M + T_P}{6}$$

关键路径法 (CPM)

这是时间管理中最重要的概念。关键路径是项目网络图中最长的路径，决定了项目的总工期。关键路径上的任何活动的延迟，都将直接导致整个项目完工日期的延迟。因此，关键路径上的所有活动的总浮动时间 (Float 或 Slack) 为零。项目经理必须对此进行最严格的监控。

关键路径法网络图示例

CPM核心概念与计算

关键路径法的计算涉及六个重要的时间参数，理解这些概念是掌握CPM的关键：

正向计算（Forward Pass）

❑ 最早开始时间 (Early Start, ES)

活动最早可以开始的时间。从项目开始节点向前计算。

ES = max(前置活动的EF)

❑ 最早完成时间 (Early Finish, EF)

活动最早可以完成的时间。

EF = ES + 活动持续时间

逆向计算（Backward Pass）

❑ 最晚完成时间 (Late Finish, LF)

活动最晚必须完成的时间，不会延误项目。从项目结束节点向后计算。

LF = min(后续活动的LS)

❑ 最晚开始时间 (Late Start, LS)

活动最晚可以开始的时间，不会延误项目。

LS = LF - 活动持续时间

浮动时间计算

❑ 总浮动时间 (Total Float, TF)

活动可以延迟而不影响项目完工日期的最大时间。

TF = LS - ES = LF - EF

关键活动的总浮动时间 = 0

❑ 自由浮动时间 (Free Float, FF)

活动可以延迟而不影响任何后续活动最早开始时间的时间。

FF = min(后续活动ES) - EF

自由浮动 ≤ 总浮动

计算步骤总结：

  1. 正向传递：从开始节点计算所有活动的ES和EF
  2. 确定项目工期：找到项目的最早完成时间
  3. 逆向传递：从结束节点计算所有活动的LS和LF
  4. 计算浮动时间：识别关键路径（TF=0的活动路径）

进度压缩技术

  • 赶工 (Crashing): 通过增加资源来缩短关键路径活动工期。这通常会增加项目成本。
  • 快速跟进 (Fast Tracking): 将原本顺序执行的活动改为并行执行。这可以缩短工期，但会增加项目的风险。

2.3 管理预算：项目成本管理

成本估算方法

  • 自下而上估算: 对WBS中每个工作包进行详细估算后汇总，最准确也最耗时。
  • 类比估算: 参照过去类似项目的实际成本，快速得出粗略估算。
  • 参数估算: 利用历史数据和统计关系计算成本，如代码行估算法、COCOMO模型。

项目预算的构成

理解预算的层级结构至关重要：

  • 对活动成本进行估算，汇总到工作包。
  • 在工作包层面加入用于"已知风险"的应急储备，形成控制账户。
  • 所有控制账户汇总，形成成本基准 (Cost Baseline)。
  • 在成本基准之上，增加用于"未知风险"的管理储备。
  • 成本基准 + 管理储备 = 项目总预算。

挣值管理 (EVM)

EVM是项目绩效测量的黄金标准，它通过集成项目的范围、进度和成本数据，提供了一个关于项目健康状况的整体视图。其威力不仅在于报告过去，更在于预测未来。

EVM 核心术语与缩写

缩写 英文全称 中文名称 含义说明
PV Planned Value 计划价值 按计划应该完成的工作的预算价值
EV Earned Value 挣值/赢得价值 实际完成工作的预算价值
AC Actual Cost 实际成本 完成工作实际发生的成本
BAC Budget at Completion 完工总预算 项目完成时的总预算
CV Cost Variance 成本偏差 EV - AC，反映成本执行情况
SV Schedule Variance 进度偏差 EV - PV，反映进度执行情况

SPI Schedule Performance Index 进度绩效指数 EV / PV，反映进度效率

ETC Estimate to Complete 完工尚需估算 完成剩余工作所需的估算成本

TCPI To-Complete Performance Index 完工尚需绩效指数 剩余工作需要达到的成本绩效

EVM 核心原理：

EVM的核心是三条曲线：PV（计划基线）、EV（实际绩效）、AC（实际花费）。通过比较这三条曲线的关系，可以同时了解项目的进度和成本状况，并预测未来趋势。

EVM 挣值分析图 (示例)

指标 (Metric) 公式 (Formula) 解读 (Interpretation)
成本偏差 (CV) $EV - AC$ > 0: 成本节约 (Under Budget)
< 0: 成本超支 (Over Budget)
进度偏差 (SV) $EV - PV$ > 0: 进度超前 (Ahead of Schedule)
< 0: 进度落后 (Behind Schedule)
成本绩效指数 (CPI) $\frac{EV}{AC}$ > 1: 资金使用效率高
< 1: 资金使用效率低（成本超支）
进度绩效指数 (SPI) $\frac{EV}{PV}$ > 1: 进度执行效率高
< 1: 进度执行效率低（进度落后）
完工估算 (EAC) $\frac{BAC}{CPI}$ 基于当前绩效预测的项目总成本 (BAC是完工总预算)
完工尚需绩效指数(TCPI) $\frac{BAC - EV}{BAC - AC}$ 为达成原始预算，剩余工作需达到的CPI

EVM 综合计算例题

例题：软件开发项目EVM分析

题目背景：

某软件开发项目计划为期8周，总预算为100万元。项目执行到第5周末时，项目经理进行了EVM分析，收集到以下数据：

  • BAC（完工总预算）：100万元
  • PV（第5周末计划价值）：80万元
  • EV（第5周末挣值）：70万元
  • AC（第5周末实际成本）：82万元

请计算以下指标：

  1. 成本偏差（CV）和进度偏差（SV）
  2. 成本绩效指数（CPI）和进度绩效指数（SPI）
  3. 完工估算（EAC）
  4. 完工尚需绩效指数（TCPI）
  5. 分析项目当前状态，并给出管理建议
点击查看详细解答过程 📝
1. 偏差计算：

成本偏差 (CV)

CV = EV - AC = 70 - 82 = -12万元

负值表示成本超支

进度偏差 (SV)

SV = EV - PV = 70 - 80 = -10万元

负值表示进度落后

2. 绩效指数计算：

成本绩效指数 (CPI)

CPI = $\frac{EV}{AC} = \frac{70}{82} =$ 0.85

小于1，成本效率低

进度绩效指数 (SPI)

SPI = $\frac{EV}{PV} = \frac{70}{80} =$ 0.875

小于1，进度效率低

3. 完工估算：

完工估算 (EAC)

EAC = $\frac{BAC}{CPI} = \frac{100}{0.85} =$ 117.65万元

预计项目总成本将超出原预算17.65万元

4. 完工尚需绩效指数：

完工尚需绩效指数 (TCPI)

TCPI = $\frac{BAC - EV}{BAC - AC} = \frac{100 - 70}{100 - 82} = \frac{30}{18} =$ 1.67

要按原预算完成，剩余工作的成本绩效需达到1.67

5. 项目状态分析与管理建议：
项目现状：
  • 项目同时存在成本超支和进度落后问题
  • 已完成工作的成本效率仅为85%，进度效率为87.5%
  • 按当前绩效趋势，项目将超支17.65万元
管理建议：
  • 立即启动纠正措施：分析成本超支和进度落后的根本原因
  • 资源优化：考虑对关键路径活动进行"赶工"或"快速跟进"
  • 成本控制：剩余工作需要将CPI提升至1.67才能按原预算完成
  • 干系人沟通：及时向管理层汇报项目风险，可能需要申请预算变更
  • 加强监控：缩短EVM报告周期，密切跟踪纠正措施效果

成本控制方法：完工尚需绩效指数（TCPI）

TCPI 成本控制原理与应用

TCPI（To-Complete Performance Index）是EVM中最重要的前瞻性指标，它告诉项目经理：为了达成特定的完工目标，剩余工作需要达到什么样的成本绩效水平。

TCPI(BAC) - 按原预算完成

TCPI = $\frac{剩余工资(BAC-EV)}{剩余资金(BAC-AC)}$

TCPI(EAC) - 按新估算完成

TCPI = $\frac{剩余工资(BAC-EV)}{剩余资金(EAC-AC)}$

TCPI 成本控制决策图

TCPI ≤ 1.0

项目仍有成本缓冲，可按原预算完成

✓ 控制良好

1.0 < TCPI ≤ 1.2

需要提高效率，但仍可实现

⚠ 需要关注

TCPI > 1.2

极难实现，需要重新规划

✗ 高风险

TCPI 实战应用策略：

  • 定期计算：在每个项目检查点计算TCPI，评估完工目标的可实现性
  • 趋势分析：跟踪TCPI的变化趋势，及早发现成本控制风险
  • 决策支持：当TCPI超过1.2时，考虑调整项目范围或申请额外预算
  • 团队沟通：向团队明确TCPI目标，确保所有人理解成本控制要求

第三部分：人员与过程职能

本部分涵盖了项目管理中"软"但同等关键的方面。一个完美的计划，如果没有一个功能健全的团队、高质量的过程和清晰的沟通渠道，终将是一纸空文。

3.1 确保卓越：项目质量管理

核心原则：质量是规划出来的

质量是规划和构建出来的，而不是在项目结束时检查出来的。这意味着质量活动必须贯穿项目的整个生命周期。其核心是理解质量成本(Cost of Quality)，即增加预防成本（培训、评审）和评估成本（测试），可以大大降低失败成本（返工、修复），从而降低总成本。

质量保证 (QA) vs. 质量控制 (QC)

质量保证 (QA)

面向过程。 通过审计和评估项目遵循的过程是否正确有效，来建立对最终产品质量的信心。回答："我们是否在 用正确的方法 做事？"

质量控制 (QC)

面向产品。 通过检查和测试具体的产出，来识别和纠正缺陷。回答："我们做出的 结果 是否正确？"

软件质量特性框架

软件质量是一个多维度的概念，不仅仅局限于"没有缺陷"。现代软件质量管理需要从产品的整个生命周期角度来考虑质量特性，包括开发、维护、转移和运行等各个阶段的质量要求。

软件质量特性三维模型

该模型展示了软件质量的三个核心维度：产品修改（可理解性、可维修性、灵活性、可测试性）、产品转移（可移植性、可重用性、互运行性）和产品运行（正确性、健壮性、效率、完整性、可用性、安全性）。

激励理论（考试重点）

核心在于理解如何激励知识型员工。重点掌握 赫茨伯格双因素理论：

  • 保健因素: 缺乏时会导致不满，但满足后并不能带来激励。例如：薪水、工作环境、公司政策。它们是基础，必须满足。
  • 激励因素: 能够真正激发员工热情和满意度的因素。例如：工作本身的挑战性、获得的认可、责任感、晋升和成长机会。

第四部分：管理不确定性与整合

4.1 驾驭未知：项目风险管理

核心概念

风险是一个不确定的事件，一旦发生，会对项目目标产生积极（机会）或消极（威胁）的影响。风险管理的核心在于主动而非被动。关键工具包括用于系统性识别风险的风险分解结构(RBS)和用于记录和跟踪所有风险的风险登记册。

定性风险分析：概率影响矩阵

通过评估每个风险发生的 概率 (Probability) 和 影响 (Impact)，对风险进行优先级排序，以决定哪些风险需要立即关注。

4.2 融会贯通：项目整体管理

项目整体管理是项目经理的核心职责，它像一根主线，将所有其他知识领域的过程和活动串联、整合在一起。

关键过程与文件

  • 制定项目章程: 这是项目的"出生证明"，正式批准项目存在、并授予项目经理权力。
  • 制定项目管理计划: 这是指导项目如何执行、监控和收尾的主文件，整合了所有子计划和基准。
  • 实施整体变更控制: 任何对项目基准的变更都必须通过正式的评估和审批流程，通常由一个名为 变更控制委员会 (CCB) 的机构来决策，从而防止无序的范围蔓延。
  • 项目收尾: 正式结束项目的活动，核心工作包括获得客户对最终成果的正式验收，以及进行行政收尾，如总结经验教训 (Lessons Learned)、归档记录和解散团队。

5.2 战略性备考指南：解构案例分析题

采用四步法：识别问题 (项目怎么了？) -> 映射知识领域 (这属于哪个管理范畴？) -> 诊断根本原因 (为什么会这样？) -> 提出解决方案 (应该怎么做？)。

例如：不要只说"项目进度落后"，而应分析说："项目进度落后（SV为负，SPI小于1），其根本原因在于关键路径上的一项活动出现了未预料的延迟，而项目计划中缺乏足够的进度应急储备来应对此类风险。建议项目经理立即对关键路径上的活动进行'赶工'，同时评估其对成本的影响，并通过整体变更控制流程获得批准。"
